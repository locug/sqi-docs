Registered Symbols & Variables

Before the dispatch loop processes any commands, the preprocessor makes a pass over the script and replaces built-in variables with their runtime values. This page covers what gets substituted and the special characters the engine uses internally.

Preprocessor Variables

The preprocessor maintains a 13-entry lookup table. When it encounters an @ followed by one of these names, it replaces the entire token with the current value:

VariableDescriptionExample Value
@TIMECurrent time (HHMMSS format)143022
@NOWCurrent date and time2026-04-01 14:30:22
@TERTerminal number901
@STOREStore number001
@USEROperator/user ID42
@USERNAMEFull usernameADMIN
@RUNRun directory pathC:\SMS\Run\
@OFFICEOffice directory pathC:\SMS\Office\
@CNCConnection number1
@PROFILEUser profile nameMANAGER
@DIDDevice IDPOS01
@DSDFCurrent date (formatted)04/01/2026
@OFFICEDBTOffice DBT directoryC:\SMS\OfficeDbt\

These are resolved before the dispatch loop runs. By the time commands like @WIZGET and @FMT are processed, all preprocessor variables have already been replaced with concrete values.

No-Args Preprocessor Commands

Three commands also appear in the preprocessor table as no-argument forms:

TokenEquivalent To
@WIZINITCalls the wizard initialization routine
@WIZDISPLAYCalls the wizard display routine
@WIZRESETClears the parameter store

When these appear without parentheses (e.g., just @WIZRESET on its own), the preprocessor resolves them directly. When they appear with parentheses (e.g., @WIZRESET()), they are handled by the dispatch loop instead.

Date Tokens (@Dxxx)

The preprocessor recognizes @D followed by format characters as inline date tokens. These let you embed formatted date components directly in your SQL:

TokenMeaningExample
@D01Day of month (2-digit)01
@DmmMonth (2-digit)04
@DyyYear (2-digit)26
@DyyyyYear (4-digit)2026
@DmmDyyMonth then year0426

The characters after @D are validated against date format character classes (digits, m, y, d), then parsed by the date formatting function to produce the current date in the requested format.

Note

Date tokens are resolved at the moment the preprocessor runs. They always reflect the current date and time when the script executes, not when it was written or loaded.

Special Characters Used During Processing

The engine uses several non-printable characters internally during processing. These never appear in .sqi source files — they exist only in memory while the engine is working.

DLE (0x10) — Escaped Open Paren / Unknown @

During the dispatch loop, when the engine encounters a command name that is not in the 68-entry table, it replaces the ( with DLE (Data Link Escape, byte 0x10). This prevents the same parenthesis from being matched again on the next iteration, avoiding an infinite loop.

During the preprocessor pass, unrecognized @ symbols are also temporarily replaced with DLE to prevent re-scanning.

DC1 (0x11) — Escaped Close Paren

The counterpart to DLE. When an unknown command is found, the ) is replaced with DC1 (Device Control 1, byte 0x11). Together with DLE, this effectively “hides” the unknown command from future matching.

Restoration

At the end of processing (Stage 4: Final Cleanup), all DLE characters are restored to ( and all DC1 characters are restored to ). The final output string contains only normal characters.

SUB (0x1A) — Exit/Success

CHR(26), the Substitute character, signals the engine to stop processing and return success. When this character appears in the string, the engine halts macro expansion and returns control to the calling application with a success state.

ESC (0x1B) — Abort/Error

CHR(27), the Escape character, signals the engine to stop processing and return an error. This is used when a script needs to abort — for example, when a validation check fails or the user cancels a dialog.

Caution

SUB and ESC are control flow mechanisms, not output characters. They are intercepted by the engine before any SQL reaches the database. If you see CHR(26) or CHR(27) in a script, it means “exit here” or “abort here.”

The Branch Delimiter Revisited

As covered in the previous page, the ® character (0xAE in Windows-1252) is used as a deferred @ inside @FMT(CMP,...) branches. There is also a second special delimiter used for inter-process communication:

CharacterPurposeWhen Converted to @
® (0xAE)Branch delimiter in CMP true/false stringsAfter CMP selects the winning branch
Exec delimiterIPC/MailSlot script commandsWhen the preprocessor detects a script starting with this character

The exec delimiter is used internally by the system when one POS terminal sends a command to another via MailSlot IPC. Scripts starting with this character are automatically wrapped in @EXEC(...) and dispatched.
