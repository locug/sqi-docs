The Branch Delimiter

The previous page showed a problem: because the engine always processes the innermost command first, macros inside conditional branches get executed before the condition is evaluated. SQI solves this with a clever character substitution trick.

The Problem

You want to conditionally set a parameter based on a mode flag:

Your intent: if MODE is 1, set RESULT to yes. Otherwise, set it to no.

What actually happens: the engine finds the first ) inside @WIZGET(MODE) and resolves it. Good. But then it finds the next ) inside @WIZSET(RESULT=yes) and resolves that too — before @FMT(CMP,...) ever runs. Both @WIZSET commands execute unconditionally, and RESULT ends up as no every time.

The Solution: Use ® Instead of @

The registered symbol ® (Unicode U+00AE, byte 0xAE in Windows-1252) acts as a deferred @. The engine does not recognize ® as a command prefix during normal dispatch. It looks like ordinary text.

How It Works Internally

The @FMT(CMP,...) handler does two things after selecting the winning branch:

  1. Returns the text of the selected branch as its result string.
  2. Scans that result string and replaces every ® with @.

The replaced text is spliced back into the main string. On the next iteration of the dispatch loop, the engine finds the now-real @WIZSET(RESULT=yes) command and processes it normally.

The discarded branch is simply thrown away. Its ® symbols are never converted, and its commands never execute.

Caution

Only use ® for commands inside branch arguments of @FMT(CMP,...). In normal (non-branching) contexts, just use @ directly. Using ® outside of CMP branches will cause the commands to never execute, because nothing will convert the ® back to @.

Nesting CMP Inside CMP

Because ® defers resolution by exactly one level, nested conditionals require stacking the technique. If you have a CMP inside a CMP branch, the inner CMP’s branches would need their own deferral mechanism. In practice, deeply nested conditionals are rare — most scripts restructure the logic across multiple statements instead.
