Preprocessor Variables

Before the main command dispatch loop runs, the preprocessor scans the entire script string for @ characters and resolves a set of 13 built-in variables by replacing them inline with their runtime values. This happens in a single pass before any @COMMAND(args) processing.

Tip

Preprocessor variables are not commands — they have no parentheses. @STORE is replaced directly, while @WIZGET(STORE) would be a command call. The preprocessor runs first, so by the time the command dispatcher sees the string, all preprocessor variables have already been resolved to concrete values.

How Preprocessing Works

  1. The preprocessor scans the script string for @ characters.
  2. For each @, it checks the text that follows against a 13-entry substitution table.
  3. If matched: the @VARIABLE text is replaced inline with the resolved value.
  4. If not matched: the @ is temporarily replaced with DLE (0x10) to prevent re-matching, then restored after the pass completes.
  5. Special @Dxxx date tokens are detected and resolved as inline date formats.

This means preprocessor variables are resolved before any commands like @WIZGET or @FMT. You cannot override them — they are always the system’s live values.

Preprocessor-Only Actions

Three entries in the preprocessor table are action triggers rather than simple variable substitutions. They have no parentheses and execute immediately when encountered:

VariableAction
@WIZINITInitialize wizard UI (no-args form)
@WIZDISPLAYDisplay wizard form (no-args form, blocks until closed)
@WIZRESETClear all keys from parameter store

Caution

These are the no-argument forms processed by the preprocessor. The command forms @WIZINIT(key=value) and @WIZRESET(prefix) (with parentheses) are handled by the command dispatcher instead. The preprocessor forms trigger different behavior.

@Dxxx Date Token Patterns

The preprocessor detects @D followed by date format characters and resolves them as inline date expressions. These produce formatted date components from the current system date.

Syntax: @Dxxx where xxx is a date format pattern

Special Characters

During processing, the engine uses special control characters internally:

CharacterCodePurpose
DLE0x10Temporarily replaces @ or ( during processing
DC10x11Temporarily replaces ) during processing
SUB0x1ACHR(26) — Exit script with success
ESC0x1BCHR(27) — Abort script with error
Branch delimiter0xAEPlaceholder for @ inside @FMT(CMP,...) branches

These are internal to the engine. The branch delimiter is the exception: it is used in .sqi source files as a placeholder for @ inside CMP branch strings.

Resolution Order

Understanding the processing order is critical for writing correct scripts:

  1. Preprocessor pass — @STORE, @TER, @TIME, @Dxxx, etc. are resolved first
  2. Command dispatch loop — @WIZGET(...), @FMT(...), @DBHOT(...) are resolved second, innermost-first
  3. Final cleanup — escaped characters are restored
